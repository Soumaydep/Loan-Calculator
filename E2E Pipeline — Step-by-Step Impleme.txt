Phase 3: Processor Lambda — Business Logic
Trigger: S3 event notification (file lands in /inbound)

Step

Action

Details

3.1

Receive S3 event & fetch file

Parse event for bucket/key → GetObjectCommand to fetch file content and metadata

3.2

Get latest batch number

QueryCommand on DynamoDB: PK = 'BATCH', ScanIndexForward: false, Limit: 1 → increment by 1 → PutCommand new record with status PROCESSING and TTL (6 months)

Parse & validate CSV

Parse file using csv-parse → validate every row against the https://docs.google.com/spreadsheets/d/1dqbKDZcR71M-SGyHXAns3gxLEI9AZe1rM4leCS9_uaQ/editConnect your Google account (required fields, data types, tran_type values, level rules)

If validation fails

Move file to /errored → generate report CSV (input + errorReason column) → upload to S3 /reports AND Google Drive /reports → update DynamoDB status to FAILED → STOP

Connect to PMM Uniprod DB

Fetch DB creds from Secrets Manager → oracledb.getConnection()

Insert records into SDIPRDMSI

Execute SDIPRDBA procedure

BEGIN SDIPRDBA; END; — wait for completion (expected < 5 min)

3.8

Fetch failed records

SELECT batch_num, prd_lvl_number, prd_lvl_id, prd_lvl_parent, prd_name_full, error_code FROM UNIPROD.SDIPRDMSI WHERE batch_num = ? — successful records are auto-deleted by the procedure

3.9

Generate report & archive

Map error_code to descriptions (05→Product Number Invalid, 08→Invalid Transaction Type, 10→Changed Record Invalid Level, 39→Null Product Name, 50→Invalid Vendor Number, 134→Invalid Style Indicator, 138→Level Not Valid) → upload report to S3 + Google Drive → move file from /inbound to /archived → update DynamoDB status: SUCCESS if 0 failures, PARTIALLY_FAILED if some failures

Phase 4: Retry Mechanism


async function withRetry<T>(fn: () => Promise<T>, maxRetries = 3, operationName: string, batchNo: number, errorCode: string): Promise<T> {
  for (let attempt = 1; attempt <= maxRetries; attempt++) {
    try { return await fn(); }
    catch (error) {
      logger.warn({ errorCode, trackingId1: batchNo, message: `${operationName} attempt ${attempt}/${maxRetries} failed: ${error.message}` });
      if (attempt < maxRetries) await sleep(Math.pow(2, attempt) * 1000); // exponential backoff
    }
  }
  // Exhausted → move file to /errored, raise P3 Datadog alert
  throw lastError;
}
Do NOT retry authorisation failures or data validation errors. After exhausting retries, move file to /errored — support can move it back to /inbound to trigger a new batch.

E2E — Happy path

Valid file → full pipeline → status SUCCESS, no report

E2E — Validation fail

Invalid file → FAILED, report in GDrive + S3

E2E — Partial fail

Valid file, some PMM rejections → PARTIALLY_FAILED, report generated

E2E — Unauthorised

File from non-authorised user → skipped, logged as PMM_PRODUCT_UPLOAD_01

E2E — Transient failure

Simulate service outage → verify 3 retries → file moved to /errored

Downstream

Confirm records in PMM via Taxonomy/Product services, reports in GDrive, Datadog alerts fire
